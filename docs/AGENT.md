# What is an Agent?

An Agent is the central coordinator of a single NPC. It owns all the state that makes that NPC a unique individual — its memories, knowledge, relationships, goals, and current tasks. The Agent never directly performs Minecraft actions, never calls the LLM directly, and never modifies world state on its own. It delegates every behavior to specialized systems and acts as the decision-making hub that routes information between them.

The Agent is not an LLM. The Agent is the entity that decides when and how to use the LLM, the memory store, the planner, and every other system.

# Agent Responsibilities

**Identity** — The NPC's name, appearance, personality traits, and core disposition. This is the immutable foundation every other system builds upon.

**Life History** — A chronological record of every significant event the NPC has experienced or observed. This is the raw material from which memory and relationships are built.

**Knowledge** — Semantic facts the NPC has learned: location of the blacksmith, the player's name, how to craft a torch, who lives in which house.

**Relationships** — Per-NPC and per-player tracked values for familiarity, trust, sentiment, and shared history. Influences conversation, trading, cooperation, and task delegation.

**Current Goal** — The high-level objective the NPC is working toward. Set by the player, by settlement membership, by survival needs, or by emergent circumstances.

**Current Plan** — A sequence of Tasks generated by the Planner to achieve the Current Goal. May be empty if no goal is active.

**Current Tasks** — The active and queued Tasks the Agent is executing right now. Drawn from the Current Plan or from immediate survival/response needs.

**Inventory View** — A read-only representation of the NPC's Minecraft inventory. The Agent can query what it holds but does not manipulate inventory directly.

**Perception** — The Agent's understanding of its immediate surroundings: nearby entities, blocks, time of day, weather, sounds, and recent events reported by the Perception system.

**Conversation State** — Tracks whether the NPC is in an active conversation, who with, the recent message history, and the current emotional tone of the interaction.

**Emotion State** — The NPC's current emotional valence (happy, sad, angry, neutral) and intensity. Influenced by events, memories, relationships, and conversation. Affects behavior priority, conversation tone, and facial expression.

**Settlement Membership** — Which Settlement (if any) the NPC belongs to, their role within it, and their current assigned duties.

# Agent Loop

The Agent runs a continuous loop, one tick per cycle:

1. **Perceive** — Gather observations from the Perception system about the current world state, nearby entities, events, and environment.

2. **Update Life History** — Append any newsworthy observations as events to the NPC's life history store. Nothing is discarded immediately; pruning happens by importance and decay.

3. **Update Knowledge** — Extract and integrate semantic facts from recent perceptions and life history entries. Update known locations, entity states, and world properties.

4. **Evaluate Goals** — Check whether the current goal is still valid, whether it has been achieved, or whether a new goal should take priority (survival, player command, settlement duty).

5. **Plan (only if needed)** — If the Current Goal has no valid Plan, invoke the Planner to generate one. The Planner may run asynchronously.

6. **Execute one Task** — Advance the next Task in the Current Tasks queue. If the Task completes, discard it and proceed to the next. If it fails, signal the Planner for replanning.

7. **Repeat** — Return to Perceive and begin the next cycle.

# Systems the Agent Owns

**Conversation** — Manages dialogue state, persona, message history, and emotion expression during player interactions. Provides the interface between the player and the Agent's internal systems.

**Life History** — Append-only, time-stamped event store. Records what the NPC saw, heard, did, and experienced. Entries have an importance score that determines retention duration.

**Knowledge** — Queryable fact store. Contains learned information about locations, entities, items, recipes, and relationships between concepts. Can be updated by perception and by conversation.

**Planner** — Given a Goal, produces a Plan (ordered list of Tasks). Runs on its own schedule, potentially asynchronous, to avoid blocking the Agent loop.

**Task Queue** — Holds the list of Tasks the Agent is currently executing. Supports sequencing, conditional branching, and prioritization. Tasks are dequeued one per loop cycle.

**Relationship Manager** — Tracks dyadic relationships between this NPC and every other entity (players and NPCs) it has interacted with. Each relationship stores familiarity, trust, sentiment, and a summary of shared history.

**Perception** — Interfaces with the Minecraft world to produce structured observations: entity sightings, block changes, time events, chat messages, damage events. Filters and prioritizes raw game events before passing them to the Agent.

**Action Executor** — Receives a Task and translates it into one or more direct Minecraft API calls. This is the only system that touches Minecraft directly. The Agent never calls the Action Executor directly — it always goes through the Task Queue.

**Settlement** — Manages the NPC's role, duties, and relationships within a Settlement. Reports settlement-level goals and resource needs to the Agent.

**Persistence** — Handles serialization and deserialization of all Agent-owned state. Triggered by world save/load events. Ensures the Agent can be paused mid-conversation, mid-task, or mid-plan and resume exactly where it left off.

# Systems the Agent DOES NOT Own

**Minecraft rendering** — The game's rendering engine, entity models, textures, and animations are outside the Agent's scope. The Agent produces intent; the client handles visual presentation.

**Networking** — HTTP communication with the AI backend, packet handling, and data synchronization between client and server are managed by a separate Networking system. The Agent sends and receives messages through it but does not manage connections.

**Entity registration** — Forge's registry system, deferred registers, and entity type registration are infrastructure concerns. The Agent consumes registered entities but does not participate in registration.

**World saving** — Minecraft's native chunk and world save system is external. The Agent's Persistence system hooks into world save events but the underlying save mechanism belongs to the game.

**Forge events** — The Forge event bus and all vanilla Minecraft events are external. The Perception system subscribes to relevant events and translates them into observations for the Agent.

# Design Rules

- The Agent never talks directly to Minecraft. Every world interaction goes through the Action Executor via the Task Queue.
- The Agent never calls the LLM directly. All LLM communication goes through the Conversation or Planner systems, which the Agent invokes.
- The Agent never modifies world state directly. It produces Tasks; the Action Executor applies them.
- The Agent always acts through Tasks. Even immediate responses (looking at a player, playing a sound) are represented as Tasks.
- The Agent always remembers Events rather than chat logs. Conversations produce memory entries about what was discussed, not verbatim transcripts.
- The Agent can be paused and resumed safely. All state is owned by the Agent and persisted through the Persistence system. Pausing mid-loop is equivalent to skipping a tick; resuming picks up from the current state.
